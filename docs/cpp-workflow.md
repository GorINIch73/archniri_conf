# C++ workflow

## Assumptions

- CMake-based projects
- build directories named `build/<BuildType>`
- `compile_commands.json` generated by CMake
- `clangd` as the language server

## New project starter

For new or personally owned C++ projects, start from:

```text
templates/cpp/.clangd
```

Copy it into the project root as `.clangd`. It enables stricter include hygiene, a small clang-tidy baseline, richer hover output, and clangd-side inlay-hint support.

## Notes

- `clang-format` is used automatically on save for C and C++ buffers; it is the main formatter for indentation and block layout.
- `cmake-tools.nvim` stores build trees separately by build type, for example `build/Debug` and `build/Release`.
- It generates `compile_commands.json` through `-DCMAKE_EXPORT_COMPILE_COMMANDS=1` and soft-links it into the project root for `clangd`.
- CMake presets take priority when a project provides `CMakePresets.json`; otherwise the normal build-type selector exposes the default variants.
- Existing projects may keep older `cmake-tools.nvim` session settings; use `:CMakeSettings` if a project does not pick up the newer build-dir layout.
- Source/header switching uses clangd's `textDocument/switchSourceHeader` extension.
- Inlay hints are off by default and can be enabled per buffer when they are useful.
- The starter `.clangd` template is intentionally opinionated; inherited or legacy projects may need a softer version, especially around Include Cleaner diagnostics.
- The DAP setup uses `gdb`, asks for the executable path when launching, and opens/closes the debug UI with the session lifecycle.
